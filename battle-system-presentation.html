Battle modes (one rules module)
Mode A — Sync

Training / vs AI / quick duel. startBattle + submitBattleAction. BATTLE_LOCK KV. Defender action from AiOpponentService (or human if you add live PvP later).

Mode B — Async PvP

startDuel / challenge. play_style recorded; battle_resolver cron simulates rounds, writes battle_round for replay, settles rewards, notifies.

Raids / revenge stay as battle types with cron + hp-manager (shields on champion or bridge health for MVP).

Currency System

XP vs spendable coins

XP is never spent. Shop spending and grants use an auditable ledger. Prefer coin_transaction + inventory_item / inventory_log for arena economy; integrate or mirror legacy transaction_history if required — do not mix XP into coin rows.

Battle Engine

Turn-Based Rule Engine

One ActionResolver + damage pipeline. Crit and defend caps are configurable per mode (sync: e.g. 15% × 1.5×; async: matrix-based 2× where product approves). All resolution server-side — no client-trusted damage. HP: primary model is per champion_instance; optional MVP bridge uses a single student arena health row for Mode A only.

⚔️
Attack
Full damage. Beats Weak Attack. Loses to Block.
🛡️
Defend
API: DEFEND. Absorbs Attack (matrix may award crit / counter). UI may read “Block”. Weak Attack bypasses. Defend limit per fight if enabled in config.
💨
Weak Attack
60% damage that bypasses Block. Loses to Attack.
Outcome Matrix
Player Action Opponent Action Result Damage Dealt
⚔️ Attack 💨 Weak Attack Win 100% DMG to opponent
🛡️ Defend ⚔️ Attack Win 0 taken + counter (see config)
💨 Weak Attack 🛡️ Defend Win 60% bypasses block
⚔️ Attack ⚔️ Attack Draw Both take 100% DMG
🛡️ Defend 🛡️ Defend Draw No damage either side
⚔️ Attack 🛡️ Defend Lose 30% counter damage taken
🎲 Crit: default for Mode A — 15% chance on qualifying wins → 1.5× (tune in battle_config). Async matrix may use cell-based 2× — unify after playtest. 15% · 1.5×

Technical Challenges

Key Problems & Solutions

🔒
Concurrent Battle Prevention
Student double-taps "Start Battle" and two battles spawn simultaneously, corrupting HP state.
Solution
BATTLE_LOCK KV namespace. Set BATTLE_LOCK[studentId] = battleId at start. Check before every action. Clear on battle end.
🎲
No-Duplicate Egg Characters
Egg system must never award a character the student already owns.
Solution
AcquisitionService: exclude owned template IDs; grant next unowned champion_instance. Fallback to bonus currency if pool empty (not only “6”).
📸
Async Opponent Consistency
Opponent student changes their squad mid-battle, changing the difficulty mid-fight.
Solution
Snapshot opponent squad stats into battle.defenderHpSnapshot at startBattle. Live changes never touch an ongoing battle.
⭐
XP vs Points Isolation
Two currency systems that must never interfere. XP only levels, Points only spend.
Solution
user_xp (or instance XP) for progression only. Spendable currency: coin_transaction + inventory_*; mirror legacy transaction_history only at integration boundary — never store XP in coin rows.
⚡
D1 CPU Time Limits
Cloudflare Workers have ~50ms CPU budget. Complex battle turns risk timeout.
Solution
Max 5 DB operations per submitAction call. Use JOINs, not N+1. Battle engine is pure TypeScript — zero I/O in resolution logic.
🛡️
Shield Item Timing
Shield must block raid/async damage while offline.
Solution
hp-manager checks shield_until on instance (or global flag). Cron respects shield before applying raid damage.
🔐
Security & fairness
Client must not dictate outcomes; students must only act on owned champions.
Solution
Server-only ActionResolver; auth checks on mutations; snapshot squad/champion at battle start; rate-limit startBattle / submitBattleAction.
